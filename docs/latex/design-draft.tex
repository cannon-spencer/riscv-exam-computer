\section{Introduction}
%=============================================================================

\subsection{Purpose / Need}

The project aims to create a \textbf{plug-and-play secure exam computer} built around a RISC-V board. The motivation comes from the growing problem of AI-assisted cheating, especially in coding exams, and the limitations of current solutions such as Honorlock, which require intrusive monitoring of student laptops.

Instead of securing arbitrary student machines, we distribute \textbf{dedicated exam devices} during tests. Each device boots into a locked-down Linux environment and launches Safe Exam Browser (SEB)~\cite{safeexambrowser} so students take exams through Canvas. Because the hardware and software stack are controlled, tampering is much harder while avoiding the privacy concerns of remote proctoring tools. The system is needed by institutions and instructors who want exam integrity without surveilling students' personal devices.

\subsection{Domain \& Prior Work}

The project sits at the intersection of secure boot, locked-down computing environments, and academic assessment. Prior work includes commercial proctoring (e.g., Honorlock, Respondus) and locked browsers (Safe Exam Browser). Our approach differs by controlling the entire device rather than layering software on untrusted student hardware.

\textbf{Current status / prior work:} This project was previously advanced by Nicholas DeSanctis. The RISC-V board already boots into a full Linux OS. Other students (Shane and Wilson) are integrating SEB with Canvas; our work will connect the RISC-V system to that pipeline.

\section{Statement of Work}
%=============================================================================

\textbf{Primary deliverable:} Connect the RISC-V system to Safe Exam Browser so a student can take an exam on the device. Target workflow: device powers on $\rightarrow$ boots into hardened Linux $\rightarrow$ automatically launches SEB $\rightarrow$ student logs into Canvas and takes the exam. A complication is that Safe Exam Browser does not officially support Linux; integration may require custom configuration or adaptation.

\textbf{Security:} Ensure the software on the device cannot be silently modified. One approach is verifying the system image with a cryptographic hash stored in secure storage; the kernel or bootloader checks the OS image at boot and refuses to boot on mismatch, forming part of a secure boot chain. The goal is a system where students cannot modify the OS or install other tools but can still complete exams, write code, and compile as needed.

\section{Stretch Goals}
%=============================================================================

(1) System for distributing configuration to exam devices (exam settings, network config). (2) Hardware-bound cryptographic keys per device (e.g., private key to decrypt/verify config), so the exam environment only runs on approved hardware. Additional hardening (kernel hardening, capability restriction, stronger secure boot) may be needed as the project progresses.
