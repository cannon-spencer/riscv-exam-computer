\section{Evidence of Soundness}
%=============================================================================

\noindent The empirical subsections below match Section~\ref{sec:testing-log} and the logs under \texttt{docs/worklog/}.

\subsection{Prior Art}

Safe Exam Browser~\cite{safeexambrowser} (ETH Z\"{u}rich) is an established locked-browser solution for Windows, macOS, and iOS, providing the Browser Exam Key (BEK) and Configuration Security Key (CSK) handshakes used by Canvas. The open-source project \texttt{seb-linux}~\cite{seblinux} provides a community Linux port targeting the same handshake protocol, with a WebKitGTK fallback backend for architectures where Qt6 WebEngine (Chromium) is unavailable. The Cage Wayland compositor~\cite{cage_kiosk} provides a proven single-application kiosk environment on Linux. These projects collectively establish that the SEB kiosk approach on Linux is viable prior to any novel work by this team.

\subsection{Theoretical Background}

Aspects of the final system are not yet implemented as production code on the RISC-V target; this subsection summarizes literature-backed foundations for those gaps.

\textbf{Exam trust model.} SEB's documented architecture~\cite{seb_concept} explains how configuration files, URL filters, and keys constrain the browser relative to the LMS. That model underpins our requirement to reproduce BEK/CSK behavior on Linux even when the official SEB build is not vendor-supported on riscv64.

\textbf{Privilege and isolation on RISC-V.} The RISC-V privileged architecture~\cite{riscv_priv} defines machine/supervisor/user modes and physical memory protection (PMP) as building blocks for enforcing boundaries between firmware, kernel, and user sessions. Our design references PMP and a verified boot path as \emph{future} hardening; the theory justifies why those mechanisms are appropriate for protecting exam configuration and reducing tampering surface, even though M2 does not complete end-to-end PMP integration.

\subsection{Empirical Evidence --- SEB Lockdown on Arch Linux}
\label{sec:empirical-arch}

\textbf{Goal:} Confirm that SEB's lockdown behavior---fullscreen enforcement and process blacklisting---functions correctly on a Linux x86 machine before targeting RISC-V hardware.

\textbf{Methodology:} SEB was installed and launched on an Arch Linux machine. The team attempted to exit the browser using standard keyboard shortcuts and window manager controls. Blacklisted applications (Discord, OBS) were also launched and SEB's response was observed.

\textbf{Results:} SEB launched successfully in fullscreen kiosk mode with no visible exit button and all standard escape shortcuts blocked. When Discord was running prior to SEB launch, SEB correctly detected it and refused to start. The same behavior was observed with OBS. These results were captured and presented in the P1 Prototype presentation (Slide~9).

\textbf{Interpretation:} The core lockdown behavior of SEB on Linux is confirmed functional. The process blacklisting mechanism works as designed. This validates the software approach and provides a reproducible baseline for subsequent RISC-V integration.

\subsection{Empirical Evidence --- Orange Pi RV Debian Boot}
\label{sec:empirical-opi}

\textbf{Goal:} Confirm that a standard Debian Linux environment can run on the target riscv64 hardware.

\textbf{Methodology:} Debian sid (riscv64) was installed on the Orange Pi RV. \texttt{apt} sources were configured to \texttt{http://deb.debian.org/debian sid main}. SSH was confirmed from lab hosts (including PuTTY on Windows at 192.168.0.61 and direct Ethernet/USB--LAN links from macOS with Internet Sharing). Package installation was verified with \texttt{apt-get}.

\textbf{Results:} The board boots successfully into Debian sid. SSH access is functional. riscv64 packages install correctly from the Debian mirrors.

\textbf{Interpretation:} The hardware is confirmed capable of running a Debian Linux environment. This establishes the foundation for the full software stack and confirms package availability for WebKitGTK and Qt6 on riscv64.

\subsection{Empirical Evidence --- seb-linux Build Attempt on riscv64}
\label{sec:empirical-seblinux}

\textbf{Goal:} Build \texttt{seb-linux} on the Orange Pi RV using the WebKitGTK fallback backend, required because Qt6 WebEngine (Chromium) has no riscv64 package support.

\textbf{Methodology:} The following packages were installed: \texttt{qt6-base-dev}, \texttt{libwebkit2gtk-4.1-dev} ($\sim$2.52.1), \texttt{libgtk-3-dev}. \texttt{qmake6} was run and correctly detected the WebKitGTK fallback, printing: \textit{``Building with WebKitGTK fallback (QtWebEngine not found or unsupported).''} \texttt{make} was then executed.

\textbf{Results:} The build failed with two compile errors in \texttt{src/browser/engines/webkitgtk/}:
\begin{enumerate}
    \item The Qt6 \texttt{signals} macro conflicted with \texttt{GDBusSignalInfo::signals} and \texttt{GtkBindingSet::signals} in \texttt{gdbusintrospection.h} and \texttt{gtkbindings.h} within \texttt{webkitgtk\_view.cpp}.
    \item \texttt{webkitgtk\_profile.cpp} passed \texttt{const char*} to \texttt{webkit\_web\_context\_set\_spell\_checking\_languages()}, which expects \texttt{const gchar* const*} (a null-terminated array).
\end{enumerate}
Both bugs were reported to the upstream maintainer via a GitHub issue. The maintainer has since claimed fixes were pushed to the repository.

\textbf{Interpretation:} The build infrastructure and dependency chain are confirmed correct---\texttt{qmake6} resolved the fallback successfully and all dependencies were satisfied. The identified bugs are shallow compile-time issues with known, targeted fixes (\texttt{\#undef signals} guard and a one-element array wrapper respectively). Full build verification on the target hardware is pending resolution of the hardware access blocker described in Section~\ref{sec:bugs}.

\section{Project State Summary}
%=============================================================================

\subsection{External Interfaces (artifact vs.\ design)}

\textbf{Demonstrated in M2 (interactive, user-visible behavior).} On x86 Linux, SEB exhibits predictable lockdown behavior: fullscreen retention, blocked escape shortcuts, and refusal to start when blacklisted applications are present (Section~\ref{sec:empirical-arch}). Those observations map to the course rubric's \emph{presentation}, \emph{perception}, and \emph{usability} expectations for the interactive path we could exercise in the lab: state is implied by fullscreen enforcement and clear failure-to-start messaging when policy is violated.

\textbf{Designed but not fully wired on RISC-V in M2.} The end-to-end student flow---device power-on, SEB launch, UF e-Learning login, Gatorlink authentication, Duo, and Canvas session---is documented as UI mockups in the R2 Design Draft (Section~4.1, Figure~1), including integrity-failure chrome (MY MACHINE / CLOSE / REVOKE). M2 establishes OS and toolchain feasibility on riscv64 (Section~\ref{sec:empirical-opi}) and isolates concrete blockers for \texttt{seb-linux} on-device (Section~\ref{sec:empirical-seblinux}); full Canvas E2E on the locked-down RISC-V image remains future work, as required by honesty in Section~\ref{sec:bugs}.

\subsection{Internal Systems}

The system is structured across four layers: Hardware (RISC-V SBC, onboard RAM, eMMC storage), OS and Boot (Linux, vendor bootloader, partitioning), Kiosk/Session (Cage Wayland compositor, process policy, shortcut lockdown), and Application (\texttt{seb-linux} / SEB-class client, Canvas exam UI). The Banana Pi deployment target uses the SpacemiT K1-class SoC~\cite{spacemit_k1}. The R2 Design Draft hardware schematic (Figure~5) summarizes expected connections: LPDDR4, eMMC, USB for keyboard/mouse, HDMI, Ethernet, GPIO, and a notional secure-engine block for future hardening.

\textbf{Communication mechanisms.} Runtime traffic to Canvas and the identity provider is \textbf{HTTPS/TLS} on the campus LAN (R2 networking diagram, Figure~3). Administrative bring-up uses \textbf{SSH} over Ethernet; \textbf{UART} is available for low-level recovery (design documentation). Protocol choices are standard; M2 focus was proving link and package viability on riscv64.

\textbf{Resilience (target vs.\ M2 reality).} \textbf{Design intent:} read-only root where possible so transient misuse is cleared on reboot, and a verified boot chain (Boot ROM $\rightarrow$ FSBL $\rightarrow$ OpenSBI $\rightarrow$ U-Boot $\rightarrow$ kernel) with signature checks consistent with the design draft. \textbf{M2 status:} Debian boot, SSH, and build attempts are demonstrated (Sections~\ref{sec:empirical-opi}--\ref{sec:empirical-seblinux}); \textbf{end-to-end measured secure boot, dm-verity-style rootfs integrity, and hardware-backed key storage are not yet demonstrated} on the prototype hardware---they remain documented targets rather than completed verification.
